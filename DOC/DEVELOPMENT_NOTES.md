# Заметки для разработчиков

## Структура файла

Первым делом при разработке и наполении нового файла необходимо придерживаться следующей структуры:

### Общие принципы

1. Код должен быть чистым и читабельным. Это означает, что он должен быть хорошо отформатирован, с понятными именами переменных и функций, а также с комментариями там, где это необходимо.
2. Логика должна быть разделена на функции и классы, чтобы облегчить понимание и поддержку кода.
3. Все функции и классы должны иметь чётко определённые обязанности и не должны быть слишком длинными или сложными.
4. Код должен быть написан с учётом возможного расширения и изменений в будущем, чтобы облегчить добавление новых функций и исправление ошибок.
5. К каждому классу и функций входящих в состав классов `Utils` должны быть добавлены комментарии формата `# --- Name of module ---` и `# --- Name of class/function ---` соответственно, чтобы облегчить навигацию по коду.
Пример:

```python
# --- Импорты ---
import some_library

# --- Утилиты ---
class Utils:
    # --- Some utility ---
    def some_utility_function(self):
        pass

# --- Иницилизация ---
def initialize():
    pass
```

6. Все функции и классы должны быть снабжены комментариями, объясняющими их назначение, входные параметры и возвращаемые значения. Это поможет другим разработчикам (или вам в будущем) быстро понять, что делает код и как его использовать.
7. Используте принципы хорошего кода YANG (You Ain't Gonna Need It) и KISS (Keep It Simple, Stupid), чтобы избегать излишней сложности и ненужных функций или классов, а также DRY (Don't Repeat Yourself), чтобы избегать дублирования кода и улучшить его поддерживаемость и SOLID (1 класс имеет 1 зону ответственности), чтобы обеспечить хорошую структуру и дизайн кода.
8. Соблюдайте PEP 8 — стиль кодирования для Python, чтобы обеспечить единообразие и читабельность кода. Это включает в себя использование отступов в 4 пробела, ограничение длины строк, использование нижнего регистра с подчёркиваниями для имен переменных и функций (но не всегда уместно, см. исключения в текущем коде), а также использование заглавных букв для имен классов.
9. Соблюдайте текущую структуру проекта и следуйте существующим шаблонам кода.

Общая структура файла должна быть следующей:

1. Импорты
2. Классы (в порядке важности, логичности и низкоуровности)
3. Функции (в порядке важности, логичности и низкоуровности)
4. Иницилизация (если есть)
5. Точка входа (если есть)

### Импорты

Они должны идти в начале и быть организованы по этому примеру:

```python
import asyncio
import logging
import pyopenGL

from cython import cdef
from . import some_module

import TESTLIB as tlib
import ANOTHERLIB as alib
```

Каждый новый импорт начинается с новой строки, а импорты из одной библиотеки группируются вместе. Импорты из стандартной библиотеки идут первыми, затем сторонние библиотеки, и в конце — локальные импорты. Сами имполты внутри блоков должны быть в алфавитном порядке. Между блоками импортов должна быть одна пустая строка.

### сlass'ы

После импортов идут определения классов. Каждый класс должен быть отделён от другого одной пустой строкой. Внутри класса методы должны быть организованы в следующем порядке:

1. `__init__` (конструктор)
2. Публичные методы (без подчёркивания в начале)
3. Приватные методы (с подчёркиванием в начале)
4. Статические методы (с декоратором `@staticmethod`)
5. Классовые методы (с декоратором `@classmethod`)

Сами же классы должны быть в порядке важности, логичности и низкоуровневости (более фундаментальные классы идут выше, более специализированные — ниже). Между методами внутри класса должна быть одна пустая строка. Сами же классы должны быть отделены друг от друга 2 пустыми строками.

### Функции

После классов идут определения функций не имеющих класса, их требуется пометить как `# --- Функции ---`. Они должны быть организованы в порядке важности, логичности и низкоуровневости (более фундаментальные функции идут выше, более специализированные — ниже). Между функциями должна быть одна пустая строка. Старайтесь не заполнять файл функциями, которые не имеют отношения к его основной цели, а также избегайте создания слишком длинных функций, которые выполняют слишком много задач. Если функция становится слишком длинной или сложной, рассмотрите возможность её разбиения на более мелкие функции с чётко определёнными обязанностями.

### Иницилизация

Если в файле есть код, который должен быть выполнен при его загрузке (например, настройка логирования, подключение к базе данных и т.п.), он должен быть помещён в отдельную функцию `initialize()`, которая будет вызываться в точке входа. Это поможет избежать выполнения кода при импорте файла в другие части проекта и обеспечит более чистую структуру кода.

### Точка входа

Если файл предназначен для запуска как самостоятельный скрипт, он должен содержать точку входа в виде блока `if __name__ == "__main__":`, который будет вызывать функцию `initialize()` и/или другие функции, необходимые для запуска программы. Это позволит запускать файл напрямую, а также импортировать его в другие части проекта без выполнения кода, предназначенного для запуска.

## Заключение

Соблюдение этих принципов и структуры поможет обеспечить чистый, читабельный и поддерживаемый код, который будет легко понимать и использовать другими разработчиками (или вами в будущем). Это также поможет обеспечить единообразие в коде и улучшить его качество, что является важным для успешной разработки проекта.
